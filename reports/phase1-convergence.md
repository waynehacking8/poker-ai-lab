# Phase 1 — CFR convergence on Kuhn Poker and Leduc Hold'em

Empirical report for the Phase 1 deliverables in
[`docs/roadmap.md`](../docs/roadmap.md). Required by
[`AGENTS.md`](../AGENTS.md) — "every experiment produces both a chart
and a Markdown report".

All numbers below come from deterministic seeded runs of the test suite
or the convergence scripts; reproduce by running the commands in
[§7](#7-reproducibility).

---

## 1. What's measured

For a joint policy `(π_0, π_1)` in a 2-player zero-sum game, the
**exploitability** is

```
expl(π) = BR_value(π_1) + BR_value(π_0)
```

i.e., how much each player could gain by best-responding to the other's
fixed policy. At a Nash equilibrium `expl = 0` (in a zero-sum game). We
compute exploitability with
[`cfr.evaluate.exploitability.best_response_value`](../cfr/evaluate/exploitability.py),
an info-set-aware two-pass best response that is linear in the game
tree size. On Kuhn it agrees with the brute-force `|A|^|I|` oracle to
1e-9 (see
[`tests/test_exploitability.py`](../tests/test_exploitability.py)).

The published Nash game values used as cross-checks:

| Game | Game value to P1 | Source |
|---|---|---|
| Kuhn Poker | −1/18 ≈ −0.0556 | Kuhn 1950 (analytical) |
| Leduc Hold'em | −0.0856 | Lanctot 2013, OpenSpiel reference |

## 2. Canonical algorithms and conventions

This report compares three algorithms in their **canonical published
forms**:

| Algorithm | Reference | Variant used here | Iter semantics |
|---|---|---|---|
| Vanilla CFR | Zinkevich 2007 | full chance enumeration | 1 iter = 1 pass over all chance outcomes |
| CFR+ | Tammelin 2014 | full chance enumeration | 1 iter = 1 pass over all chance outcomes × 2 traversers |
| MCCFR | Lanctot 2009 | External Sampling | 1 iter = 1 sampled chance outcome × 1 traverser |

OpenSpiel's `CFRSolver` and `CFRPlusSolver` classes both use full
chance enumeration as the canonical form. MCCFR is sampling by
definition.

The package also ships **chance-sampling variants** of vanilla CFR and
CFR+ (`vanilla_cfr.train`, `cfr_plus.train`) — these are practical
alternatives for memory-constrained settings, not literature-canonical
algorithms. They are documented separately in §6 and not part of the
headline comparison.

## 3. Kuhn Poker

12 information sets total (6 per player). The smallest non-trivial
imperfect-information game — used as a unit test for any CFR
implementation.

### 3.1 Canonical convergence (deterministic enum)

Pinned by [`tests/test_vanilla_cfr_enum.py`](../tests/test_vanilla_cfr_enum.py)
and [`tests/test_cfr_plus_enum.py`](../tests/test_cfr_plus_enum.py):

| Algorithm | Iter | Wall time | Expl |
|---|---|---|---|
| Vanilla CFR (enum) | 50 | 5 ms | 0.054 |
| Vanilla CFR (enum) | 200 | 18 ms | 0.024 |
| Vanilla CFR (enum) | 1 000 | 92 ms | 0.013 |
| Vanilla CFR (enum) | 5 000 | 450 ms | 0.0047 |
| CFR+ (enum) | 50 | 7 ms | 0.0056 |
| CFR+ (enum) | 200 | 27 ms | 0.00059 |
| CFR+ (enum) | 1 000 | 130 ms | 0.00018 |
| CFR+ (enum) | 5 000 | 680 ms | 0.000055 |

At equal iter count CFR+ is **~75× better** than vanilla on Kuhn —
the order-of-magnitude acceleration Tammelin 2014 reports. Game value
converges to Kuhn's analytical -1/18 ≈ -0.0556 within 0.001 at 1k iter.

### 3.2 MCCFR — sampling-based

| Algorithm | Iter | Expl (5-seed mean ± std) |
|---|---|---|
| MCCFR ES | 50 000 | 0.0136 ± 0.0071 |

MCCFR ES converges to Nash but with higher per-iter variance because
chance, the non-traverser's hand, and the opponent's action are all
sampled. Per-iter cost is much cheaper than enum (one path, not 6
deals) but the iter count needed is much higher on this small game.

## 4. Leduc Hold'em

528 information sets (264 per player), 2 betting rounds, capped at 2
raises per round. The canonical small benchmark for CFR research
(Lanctot 2013, OpenSpiel).

### 4.1 Canonical convergence (deterministic enum)

Pinned by [`tests/test_vanilla_cfr_enum.py`](../tests/test_vanilla_cfr_enum.py)
and [`tests/test_cfr_plus_enum.py`](../tests/test_cfr_plus_enum.py):

| Algorithm | Iter | Wall time | Expl | Game value |
|---|---|---|---|---|
| Vanilla CFR (enum) | 50 | 2.7 s | 0.221 | -0.089 |
| Vanilla CFR (enum) | 200 | 11 s | 0.073 | -0.083 |
| Vanilla CFR (enum) | 500 | 28 s | 0.057 | -0.080 |
| Vanilla CFR (enum) | 1 000 | 56 s | 0.033 | -0.079 |
| **CFR+ (enum)** | 50 | 4.3 s | **0.140** | -0.082 |
| **CFR+ (enum)** | 200 | 17 s | **0.015** | -0.078 |
| **CFR+ (enum)** | 500 | 43 s | **0.0040** | -0.078 |
| **CFR+ (enum)** | 1 000 | 85 s | **0.0010** | -0.078 |

CFR+ vs vanilla on Leduc — the headline Tammelin 2014 result:

| Iter | Vanilla expl | CFR+ expl | CFR+ advantage |
|---|---|---|---|
| 50 | 0.221 | 0.140 | 1.6× |
| 200 | 0.073 | 0.015 | 4.8× |
| 500 | 0.057 | 0.0040 | 14× |
| 1 000 | 0.033 | 0.0010 | **35×** |

The advantage grows with iteration count — exactly the "order-of-
magnitude faster" Tammelin reports. At equal wall time, CFR+ reaches
~10× lower expl on Leduc than vanilla CFR.

Game value tracks Lanctot 2013's Nash value (-0.0856) within 0.01 at
every iteration count ≥ 50.

### 4.2 MCCFR — sampling-based

Pinned by [`tests/test_leduc_mccfr.py`](../tests/test_leduc_mccfr.py).

| Iter | Wall time | Expl |
|---|---|---|
| 5 000 | 0.6 s | 1.55 |
| 50 000 | 4.1 s | 0.54 |
| 200 000 | 16.5 s | 0.47 |
| 500 000 | 41.8 s | 0.28 |

MCCFR ES needs orders-of-magnitude more iterations than enum variants
to reach the same expl on Leduc. **This is expected**:

- MCCFR is designed for **asymptotic advantage on large games** (Lanctot
  2009 §1, §6). When the full tree is too big to traverse, sampling a
  single trajectory per iteration is cheaper than 120-deal enumeration
  by a factor of |deals| × depth. The wall-time tradeoff flips in
  MCCFR's favour at NLHE scale.
- On a 528-info-set game like Leduc, every traversal is cheap and
  full enumeration is the better choice. MCCFR's per-iter variance
  doesn't get amortised over enough info-set visits to pay off.

## 5. Cross-game observations

1. **Canonical ordering on small games:** CFR+ enum < Vanilla CFR enum
   < MCCFR. This matches Tammelin 2014, Zinkevich 2007, and Lanctot 2009
   as a unified picture. MCCFR's advantage is asymptotic — it pays off
   on large games where full enumeration is infeasible, not on toy
   games like Kuhn / Leduc.

2. **Two-pass best response scales.** Brute-force best response is
   `|A|^|I|` — infeasible past Kuhn's 64 pure strategies. The two-pass
   algorithm replaces this with a single tree DFS plus a depth-sorted
   info-set pass, linear in tree size. On Leduc this finishes in well
   under a second; the brute-force enumerator would never terminate.

3. **Strategy averaging is what converges.** All three algorithms make
   their *last-iterate* strategies oscillate; the time-averaged
   `strategy_sum` is the Nash approximation. CFR+ accelerates this by
   weighting iteration `t` with weight `t` in the average.

4. **Sign convention.** All three algorithms follow the Neller & Lanctot
   2013 convention: `cfr()` returns the value to the current player at
   `history`; recursive calls negate **only when** the next node's
   current player differs (see
   [`cfr/algorithms/vanilla_cfr.py:84`](../cfr/algorithms/vanilla_cfr.py)).
   Leduc's `'/'` round separator can leave the same player on the clock
   across a round boundary, so the naïve `len(history) % 2` parity
   trick is unsafe.

## 6. Chance-sampling alternatives (non-canonical hybrids)

The package also exposes `vanilla_cfr.train` and `cfr_plus.train` —
chance-sampling variants that visit one deal per iteration instead of
all of them. These are *not* the canonical algorithm as presented in
Zinkevich 2007 / Tammelin 2014, and they are not what OpenSpiel's
solvers use; they are practical alternatives for settings where full
chance enumeration is itself expensive.

On the toy games here, full enumeration is cheap enough that the
chance-sampling variants offer no real advantage:

All numbers in this table are seed=42 single-seed runs on the same
laptop; chance-sampling rows have small seed-to-seed variance which
this presentation does not show.

| Variant | Game | Iter | Wall | Expl |
|---|---|---|---|---|
| `vanilla_cfr.train` (chance sample) | Leduc | 50 000 | 22.5 s | 0.074 |
| `vanilla_cfr.train_enumeration` | Leduc | 500 | 27.5 s | 0.057 |
| `cfr_plus.train` (chance sample) | Leduc | 20 000 | 22.4 s | 0.154 |
| `cfr_plus.train_enumeration` | Leduc | 200 | 16.6 s | 0.015 |

At similar wall time, enumeration reaches lower exploitability on
both algorithms (vanilla: 0.057 vs 0.074 in roughly the same 25 s
window; CFR+: 0.015 vs 0.154 — enum is **10× better** for less wall
time). For games of Leduc scale or smaller, **always prefer
enumeration**.

### Bug fix history

An earlier draft of `cfr_plus.train_enumeration` had two bugs that
silently slowed convergence by ~100× on Leduc, identified after
cross-checking against OpenSpiel's reference implementation:

1. **Inline RM+ floor.** The implementation clipped regrets to 0
   inline during traversal. With 120 chance outcomes enumerated per
   iteration, later visits read the just-floored regrets — corrupting
   the accumulator. OpenSpiel's source spells out the rule:
   *"This must be done at the level of the information set, and thus
   cannot be done during the tree traversal."* Fix: accumulate
   regrets plainly during traversal, apply the floor as a separate
   step after each player's full pass.

2. **Live strategy reading.** The strategy at each info set was
   recomputed from `regret_sum` at every visit. Since regrets update
   within the traversal, downstream info-set strategies drifted under
   the same iteration. Canonical CFR+ keeps the strategy snapshot
   frozen for the duration of a player's traversal. Fix: snapshot
   strategy at iteration start, pass it through the recursion.

Before / after the fix (Leduc):

| Iter | Buggy expl | Fixed expl |
|---|---|---|
| 200 | 0.21 | 0.015 |
| 1 000 | 0.14 | 0.0010 |

Both bugs were discovered because the buggy ordering put CFR+
*slower* than vanilla CFR on Leduc — a counter-intuitive result that
contradicted Tammelin 2014. The lesson: if a result contradicts the
literature, treat it as a bug to find, not a property to defend.

## 7. Limitations

- **Toy games only.** Texas Hold'em is out of scope by design decision
  [D2](../docs/design-decisions.md). Nothing in this report extrapolates
  past 528 info sets.
- **MCCFR-only multi-seed bands.** Enum variants are deterministic
  (no RNG) so a single run is the population. MCCFR is plotted with
  3 seeds on Leduc and 5 seeds on Kuhn; bands are < 7 % of mean
  width, so more seeds would not materially change the chart.
- **CPU only.** Wall-time numbers are single-threaded NumPy on a
  laptop. The GPU acceleration story lives in
  [`docs/flashcfr-phase1-design.md`](../docs/flashcfr-phase1-design.md)
  and is not measured here.

## 8. Reproducibility

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt pytest

# Run all Phase 1 tests (fast + slow gates).
pytest tests/test_kuhn_cfr.py tests/test_vanilla_cfr_enum.py \
       tests/test_mccfr.py tests/test_leduc_mccfr.py \
       tests/test_cfr_plus.py tests/test_cfr_plus_enum.py \
       tests/test_leduc_cfr.py tests/test_leduc_cfr_plus.py

# Regenerate the convergence plots used in this report.
python -m scripts.plot_convergence_kuhn
python -m scripts.plot_convergence_leduc

# Smoke-test entry points for ad-hoc runs.
python -m scripts.smoke_test_kuhn 200000
python -m scripts.smoke_test_leduc 50000
```
